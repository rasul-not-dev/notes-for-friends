## 1. Настройка профиля (Конфигурация)

Эти команды выполняются один раз сразу после установки Git. Они нужны, чтобы Гит подписывал твои сохранения твоим именем.

- **`git config --global user.name "Имя"`** — Задать твое имя разработчика.
    
- **`git config --global user.email "email@example.com"`** — Задать твою рабочую почту.
    
- **`git config --global --list`** — Показать список всех настроек, которые сейчас применены в системе.
    

> **Продвинутая фишка (Алиасы):** Ты можешь создавать короткие псевдонимы для длинных команд. Например, сделать так, чтобы вместо `git init` можно было писать просто `git i`: `git config --global alias.i init`

## 2. Локальная работа с файлами

Классический цикл создания проекта и сохранения его на твоем компьютере.

- **`git init`** — Включает Git в текущей папке (превращает обычную папку в репозиторий).
    
- **`git status`** — Показывает статус файлов: какие из них новые (**untracked**), какие изменены (**modified**), а какие уже готовы к сохранению.
    
- **`git add [имя_файла]`** — Добавляет конкретный файл на «сцену» / в черновик (Staging Area).
    
- **`git add .`** — Берет **вообще все** измененные и новые файлы в папке и разом отправляет их в черновик.
    
- **`git commit -m "Сообщение"`** — Запечатывает файлы из черновика и сохраняет их в историю проекта как полноценную версию (снимок).
    

## 3. Выход в облако (Связь с GitHub)

Эти команды используются, когда тебе нужно работать с удаленным сервером (сайтом GitHub).

- **`git clone [ссылка_на_репозиторий]`** — Скачивает (клонирует) готовый проект с GitHub к тебе на компьютер со всей его историей.
    
- **`git push origin [имя_ветки]`** — Отправляет (выталкивает) твои локальные коммиты с компьютера на сайт GitHub в указанную ветку.
    
- **`git pull`** — Скачивает (стягивает) свежие изменения с сайта GitHub и автоматически обновляет файлы у тебя на компьютере.
    
    > _Важно:_ Всегда делай `git pull`, если кто-то (или ты сам через сайт) обновил главную ветку в облаке, иначе твой компьютер останется со старой версией кода.
    

4. Работа с ветками (Branches)

Ветки позволяют делать новые фичи изолированно, не ломая основной рабочий код в ветке `main`.

- **`git checkout -b [имя_новой_ветки]`** — Создает новую ветку и автоматически переключает тебя на нее.
    
- **`git branch`** — Выводит список всех веток в проекте (текущая ветка будет подсвечена).
    
- **`git switch [имя_ветки]`** — Переключает тебя на уже существующую ветку (например: `git switch main`).
    

## 5. Инспекция истории и удаление

Команды для проверки того, что сделано, и удаления временных веток после окончания работы.

- **`git show`** — Показывает подробнейшую информацию о самом последнем коммите: автора, дату, сообщение и **diff** (какие конкретно строчки кода были добавлены, а какие удалены).
    
- **`git branch -d [имя_ветки]`** — Удаляет ветку **локально** на твоем компьютере (делать это нужно, только когда ветка больше не нужна).
    
- **`git push origin --delete [имя_ветки]`** — Удаляет ветку **удаленно** на самом сайте GitHub.
    

## Идеальная цепочка действий (GitHub Flow)

Вот как выглядит правильный цикл работы над задачей на чемпионате или в команде:

1. Заходим в проект ➔ обновляем его до последней версии: **`git pull`**.
    
2. Создаем ветку под свою задачу: **`git checkout -b feature-buttons`**.
    
3. Пишем код в редакторе.
    
4. Проверяем себя: **`git status`**.
    
5. Добавляем изменения: **`git add .`**.
    
6. Сохраняем коммит: **`git commit -m "Добавил кнопки на главный экран"`**.
    
7. Отправляем на сайт: **`git push origin feature-buttons`**.
    
8. На сайте GitHub открываем **Pull Request** ➔ вливаем код в `main`.
    
9. Возвращаемся в терминале на `main` (**`git switch main`**) и забираем обновленный код (**`git pull`**).
    
10. Удаляем старую ветку-черновик.
