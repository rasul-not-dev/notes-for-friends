## 🛑 Уровень 1: Ошибка в коде (Коммит еще НЕ сделан)

_Ты написал плохой код в редакторе, запутался и хочешь вернуть файл к состоянию, которое было в последнем успешном коммите._

- **`git restore <имя_файла>`** — Стирает все незакоммиченные изменения в конкретном файле и возвращает его к состоянию последнего коммита.
    
- **`git restore .`** — То же самое, но сбрасывает изменения **во всех** файлах проекта разом.
    
- _(В старых инструкциях вместо этой команды использовали `git checkout -- <имя_файла>`, это то же самое)._
    

## 💾 Уровень 2: Ошибка на «Сцене» (Сделал `git add`, но НЕ коммитил)

_Ты случайно добавил в черновик (Staging Area) лишний файл (например, огромную папку `node_modules` или секретные пароли) и не хочешь, чтобы он попал в коммит._

- **`git rm --cached <имя_файла>`** — Убирает файл со «сцены» (из индекса), делая его снова серым/красным (`untracked`). При этом сам файл с компьютера **не удаляется**.
    
- **`git restore --staged <имя_файла>`** — Современный аналог предыдущей команды. Просто убирает файл из черновика будущего коммита.
    

## 📝 Уровень 3: Ошибка в самом последнем коммите (НЕ отправлен в облако)

_Ты нажал Enter на команде `git commit`, но тут же понял, что совершил ошибку (сделал опечатку в тексте или забыл прикрепить файл). Изменения пока только у тебя на ПК._

### Вариант А: Опечатка в тексте комментария

- **`git commit --amend -m "Новый правильный текст"`** — Переписывает сообщение твоего самого последнего коммита. Никаких новых «точек сохранения» не создается, старый плохой коммит просто заменяется правильным.
    

### Вариант Б: Мягкий откат (Хочу дописать код)

- **`git reset --soft HEAD~1`** — **Самая полезная команда!** Она «распечатывает» твой последний коммит. Сам коммит исчезает из истории, но весь твой написанный код **остается на месте** и возвращается на «сцену» (в состояние `git add`). Ты можешь дописать файлы и закоммитить всё заново.
    

### Вариант В: Жесткий откат (Удалить всё к чертям)

- **`git reset --hard HEAD~1`** — 🔥 **Опасная команда!** Полностью стирает последний коммит вместе со всем кодом, который ты там написал. Файлы на компьютере мгновенно возвращаются к состоянию _предпоследнего_ коммита. Используй, только если код реально мусорный и тебе его не жалко.
    

## 🌐 Уровень 4: Ошибка уже улетела на GitHub (Сделал `git push`)

_Самый ответственный момент. Если код уже в облаке, использовать `git reset` нельзя, иначе у твоих коллег или экспертов на чемпионате сломается история репозитория._

- **`git revert <хэш_коммита>`** — Создает **новый коммит-противоядие**. Он не стирает старый плохой коммит, а делает противоположные действия (если ты добавил строку — он её удалит). История проекта остается целой, а баг исправлен.
    
    - _Как узнать хэш?_ Напиши `git log --oneline`, скопируй 7 символов плохой версии и вставь: например, `git revert a1b2c3d`.
        
- **Кнопка Revert на GitHub** — Если плохая ветка уже была слита с `main` через сайт, зайди в этот Pull Request на GitHub и нажми зеленую/серую кнопку **Revert**. Сайт сам создаст обратный PR, тебе останется только подтвердить его.
    

## 🧠 Краткая шпаргалка-выручалочка:

| **Что случилось?**               | **Какую команду вводить?**      | **Что произойдет?**                            |
| -------------------------------- | ------------------------------- | ---------------------------------------------- |
| Накосячил в коде до коммита      | `git restore .`                 | Файлы станут чистыми, как в последнем коммите. |
| Добавил в индекс лишний файл     | `git restore --staged <файл>`   | Файл уберется из будущего коммита.             |
| Опечатался в коммите             | `git commit --amend -m "текст"` | Текст коммита заменится на новый.              |
| Сделал плохой коммит локально    | `git reset --soft HEAD~1`       | Коммит удалится, но код вернется в черновик.   |
| Отправил плохой коммит на GitHub | `git revert <хэш>`              | Создастся новый коммит, отменяющий старый.     |
