
Этот урок посвящен тому, как Гит физически разделяет «параллельные вселенные» на твоем компьютере, и как правильно переносить изменения.

### 🎭 Магия переключения (Как файлы исчезают и появляются)

Когда ты переключаешься между ветками, Гит буквально перекраивает твою рабочую папку в Windows.

- **Ветка А (`master`):** Здесь лежит твой стабильный сайт.
    
- **Ветка Б (`feature-login`):** Здесь ты пилишь форму входа, создал файл `login.html` и сделал коммит.
    

**Что происходит в проводнике:**

1. Находясь в `feature-login`, ты видишь файл `login.html`.
    
2. Пишешь `git checkout master` — файл `login.html` **мгновенно исчезает** из папки на компе. Гит стер его, чтобы вернуть состояние ветки `master`.
    
3. Пишешь `git checkout feature-login` — файл **магически появляется** на том же месте со всем кодом.
    

> ⚠️ **Главный закон переключения:** Перед тем как прыгнуть на другую ветку, ты обязателен должен закоммитить (или временно спрятать) свои текущие изменения. Если у тебя болтаются недоделанные незакоммиченные файлы, Гит либо не разрешит переключиться, либо перетащит этот «мусор» за тобой в другую ветку.

### 🛠️ Пошаговый рабочий процесс (Workflow) соло-разработчика

В видео показывают стандартный цикл создания фичи, который должен войти у тебя в привычку:

1. **Создаем ветку под задачу и прыгаем в нее:**
    
    Bash
    
    ```
    git checkout -b add-dark-mode
    ```
    
2. **Пишем код, создаем файлы.** (Например, настраиваем темную тему оформления).
    
3. **Делаем коммиты по ходу дела:**
    
    Bash
    
    ```
    git add .
    git commit -m "Добавил темные стили для главной страницы"
    ```
    
    _(Ветка `add-dark-mode` ушла вперед, а ветка `master` спокойно отдыхает в прошлом)._
    
4. **Возвращаемся в базу:** Когда фича готова, ты переключаешься обратно:
    
    Bash
    
    ```
    git checkout master
    ```
    
    _(Твой код с темной темой «спрятался», ты снова в чистом рабочем проекте)._
    

### 👁️ Как посмотреть это дерево ветвлений в консоли?

Чтобы не запутаться, где какая ветка находится, используется расширенная команда логов. Она рисует текстовый граф (схему) прямо в консоли:

Bash

```
git log --oneline --graph --all
```

- `--graph` — рисует палочки и звездочки (визуальное дерево).
    
- `--all` — показывает историю **всех** веток сразу, а не только той, в которой ты сидишь.
    

В консоли ты увидишь что-то вроде:

Plaintext

```
* 7b3a1c2 (HEAD -> add-dark-mode) Добавил темные стили
| * 1a2b3c4 (master) Исходный рабочий сайт
|/
* f7e6d5c Родоначальный коммит, где ветки разделились
```

### 💡 Вывод для Обсидиана

Ветки — это не копия проекта, это **разные состояния одной и той же папки**. Гит сам подменяет файлы на твоем диске в зависимости от того, куда указывает `HEAD`. Ты можешь создавать сколько угодно веток, переключение между ними безопасно, если все изменения закоммичены.





**ШПАРГАЛКА**

- `git checkout -b <имя_ветки>` — создать ветку под задачу и сразу перейти в нее.
    
- `git add .` + `git commit -m "сообщение"` — закоммитить изменения перед любым переключением.
    
- `git checkout master` (или `main`) — вернуться в главную ветку (код из побочной ветки временно «исчезнет» из папки).
    
- `git log --oneline --graph --all` — посмотреть красивое дерево всех веток и коммитов прямо в консоли.
    

**Суть на пальцах:** Git физически меняет файлы в твоей папке в Windows при переключении веток. Файлы то появляются, то исчезают — это нормально, Git так управляет состояниями. **Главный закон:** перед переключением на другую ветку обязательно делай коммит, иначе Git либо не пустит, либо перетащит незакоммиченный «мусор» с собой.
