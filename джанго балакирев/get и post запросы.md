## 1. GET- и POST-запросы (Передача данных на сервер)

### Что такое GET-параметры?

Это дополнительные параметры, которые передаются прямо в адресной строке браузера (в URL-адресе).

- **Как их узнать в URL:** Они всегда идут после специального стартового символа — знака вопроса (`?`).
    
- **Формат:** Данные передаются в виде пар `ключ=значение`. Если параметров несколько, они разделяются знаком амперсанда (`&`).
    
- _Пример из видео:_ `[http://127.0.0.1:8000/cats/music/?name=gagarin&type=pop](http://127.0.0.1:8000/cats/music/?name=gagarin&type=pop)` — здесь передаются два параметра: `name` (со значением `gagarin`) и `type` (со значением `pop`).
    

### Как поймать GET-параметры в Django?

В `urls.py` для них ничего прописывать не нужно. Django автоматически упаковывает их в специальный словарь `request.GET`. Во `views.py` их можно прочитать или вывести в консоль:

Python

```
def categories_by_slug(request, cat_slug):
    # Проверяем, если словарь параметров не пустой, выводим его в консоль PyCharm
    if request.GET:
        print(request.GET) # В консоли отобразится: {'name': ['gagarin'], 'type': ['pop']}
        
    return HttpResponse(f"Раздел: {cat_slug}")
```

### Что такое POST-запросы?

Они обычно используются в веб-программировании для передачи данных из заполненных форм (например, отправка анкеты).

- В Django данные таких запросов ловятся точно так же, но через другую коллекцию — **`request.POST`**.
    

## 2. Кастомная страница ошибки 404 (Обработка исключений)

Когда пользователь пытается открыть страницу по несуществующему URL-адресу, возникает исключение 404 (Страница не найдена). По умолчанию во время разработки Django показывает желтую страницу с технической отладкой, но для реальных пользователей нужно делать красивую кастомную страницу.

### Подготовка: Отключение режима отладки

Кастомные страницы ошибок работают **только** тогда, когда сайт запущен в «боевом» режиме.

1. В файле `settings.py` меняем `DEBUG = True` на **`DEBUG = False`**.
    
2. В список разрешенных хостов `ALLOWED_HOSTS` обязательно добавляем наш локальный адрес: `ALLOWED_HOSTS = ['127.0.0.1']` (иначе сервер выдаст ошибку доступа).
    

### Шаг 1. Создание функции-обработчика во `views.py`

Пишем функцию, которая будет красиво отвечать пользователю при ошибке 404. Она **обязательно** должна принимать два аргумента: `request` и `exception`.

Python

```
from django.http import HttpResponseNotFound

def page_not_found(request, exception):
    # Возвращаем объект специального класса HttpResponseNotFound,
    # чтобы сервер отдал клиенту честный статус-код 404 (страница не найдена)
    return HttpResponseNotFound("<h1>Страница не найдена</h1>")
```

### Шаг 2. Регистрация обработчика в главном `urls.py`

Чтобы Django узнал о нашей функции, её нужно подключить в **главном** файле `urls.py` проекта (в пакете конфигурации сайта) через специальную переменную **`handler404`**:

Python

```
# Из приложения women импортируем нашу функцию
from women.views import page_not_found 

# Присваиваем переменной handler404 ссылку на эту функцию
handler404 = page_not_found
```

теперь при любом несовпадении URL-адресов Django автоматически вызовет нашу функцию `page_not_found`.

## 3. Искусственный вызов ошибки 404 из кода (`raise Http404`)

Иногда URL-адрес верный, но логика программы требует показать ошибку (например, запрашиваемые данные отсутствуют). Мы можем вручную активировать механизм 404 ошибки в любом месте программы с помощью класса `Http404`.

_Пример использования во `views.py`:_

Python

```
from django.http import Http404, HttpResponse

def archive(request, year):
    # Если переданный год больше 2023, мы искусственно вызываем ошибку
    if year > 2023:
        raise Http404() # Мгновенно сработает наш handler404 и покажет кастомную страницу
        
    return HttpResponse(f"Архив по годам: {year}")
```

### Другие системные обработчики ошибок в Django:

Помимо 404, в главном `urls.py` можно переопределить и другие переменные для обработки исключений:

- `handler400` — если невозможно обработать запрос (Bad Request).
    
- `handler403` — доступ к странице запрещен (Permission Denied).
    
- `handler500` — внутренняя ошибка самого сервера (Server Error).
